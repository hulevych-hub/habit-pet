# EVENTS

## Overview

Habit Pet uses an event-driven architecture to communicate between different game systems. Events are used to decouple components, trigger UI updates, generate journal entries, award rewards, and track player progress. The game implements multiple event systems for different purposes.

## Current Implementation

The event system consists of:
1. **RewardUiEvent** - Sealed class for reward popup events
2. **AchievementEngine** - Monitors game state and unlocks achievements
3. **StreakEngine** - Evaluates streaks and awards chest rewards
4. **Reward Event Bus/Queue System** - Manages event prioritization and display
5. **ActivityTimelineEngine** - Persists gameplay events to the activity timeline
6. **Various ViewModel callbacks** - For UI state updates

## Rules

### RewardUiEvent System
The primary event system for rewards and popups:

**Event Types**:
- `CoinReward(amount: Int)` - Awards coins directly
- `LevelUpReward(previousLevel: Int, level: Int, coins: Int)` - Celebrates a level-up; `RewardManager` applies the level-up coins
- `DragonEvolutionReward(fromStage: Int, toStage: Int)` - Signals evolution (0 coins)
- `StreakReward(streak: Int, coins: Int, rewardSummary: List<String>)` - Shows a global streak celebration before the chest reward
- `ChestReward(rewardType: String, amount: Any, expAmount: Int, customizationId: Long?, equipableId: String?)` - Awards chest contents and can include EXP/customization
- `ExpReward(amount: Long)` - Awards XP through `RewardManager`, including level/evolution recalculation
- `CustomizationReward(equipableId: String)` - Grants a configured customization item

Achievement claims no longer use `RewardUiEvent.AchievementReward`; `AchievementRewardProcessor` prepares individual reward events so the centralized reward pipeline can apply them once.

**Event Flow**:
1. Events are generated by various engines and factories (`AchievementEngine`, `StreakEngine`, `ChestRewardFactory`, `HabitDetailViewModel`)
2. Events are added to RewardQueue via `rewardQueue.addReward(event)`
3. RewardQueue prioritizes events and emits them one at a time
4. RewardManager processes events, applies rewards, extracts coin values, and updates progression state
5. UI layer collects events and displays appropriate popups/screens

**Event Priority** (in RewardQueue):
1. LevelUpReward (priority 1 - highest)
2. DragonEvolutionReward (priority 2)
3. StreakReward (priority 3)
4. ChestReward (priority 4)
5. ExpReward (priority 5)
6. CustomizationReward (priority 6)
7. CoinReward (priority 7 - lowest)

### Legacy Journal Data
`JournalEntryEntity` remains in the Room database for legacy compatibility, but the active journal generation system, journal repository, journal DAO, and journal screen were removed. Significant milestone history is now represented by `GameEventEntity` through `ActivityTimelineEngine`.

### ActivityTimelineEngine Events
`ActivityTimelineEngine` persists gameplay moments into the activity timeline.

**Persistent Event Types**:
- `HABIT_COMPLETED`
- `ACHIEVEMENT_UNLOCKED`
- `LEVEL_UP`
- `DRAGON_EVOLUTION`
- `CHEST_OPENED`
- `STREAK_MILESTONE`
- `FIRST_DAILY_LOGIN`

**Event Flow**:
1. Gameplay systems call `ActivityTimelineEngine` or emit `RewardUiEvent.ChestReward` through `RewardEventBus`.
2. `ActivityTimelineEngine` builds `GameEventEntity` records with `GameEventFactory`.
3. `GameEventRepository` appends events through `GameEventDao`.
4. `ActivityTimelineViewModel` loads events in reverse chronological order for `ActivityTimelineScreen`.

### StreakEngine Events
Evaluates habit completion streaks and awards rewards:

**Trigger Conditions**:
- When all habits are completed on a given day
- Checks the highest unrewarded global streak milestone where currentStreak reached 7, 14, 30, 60, or 100 days and currentStreak > lastStreakAwardedAt

**Events Generated**:
- `StreakReward` with the milestone streak and reward summary, followed by `ChestReward`
- `STREAK_MILESTONE` timeline event through `ActivityTimelineEngine`
- `ChestReward` rewardType uses `global_streak_{milestone}_{chestType}`
- Chest type mapping: 7 = Normal, 14 = Rare, 30/60 = Epic, 100 = Legendary
- Updates StatisticsEntity.lastStreakAwardedAt to the awarded milestone

### Additional Event Systems

**ViewModel State Updates**:
- HabitDetailViewModel emits:
  * `HABIT_COMPLETED` timeline event when checkbox/timer habit completed
  * `LEVEL_UP` and `DRAGON_EVOLUTION` timeline events when XP crosses thresholds
  * _habitCompleted SharedFlow (Unit) when checkbox/timer habit completed
  * _rewardUiEvent SharedFlow (RewardUiEvent) for various rewards
  * _navigateBack SharedFlow (Unit) to navigate back after completion
- HomeScreenViewModel combines multiple states into UiState for UI consumption

**Database Change Events**:
- Room database flows notify observers of entity changes
- Used by ViewModels to update UI state when data changes

## Configuration

Reward event values are centralized in configuration classes rather than duplicated across ViewModels:
- RewardUiEvent class definitions are sealed in RewardUiEvent.kt
- AchievementEngine thresholds live in `AchievementsConfig`
- StreakEngine global streak milestones are 7, 14, 30, 60, and 100 days
- Chest contents are built by `ChestRewardFactory` using `ChestRewardConfigProvider` and `EconomyConfig`
- Level-up coins are `level Ă— 10` through `ExpConfig.levelUpCoins(...)` and are applied by `RewardManager`
- Level-up chests are randomized through `ChestRewardConfigProvider.getRandomChestType()`
- Timer habit coins are `5 base coins + 2 coins per accumulated minute`, calculated from `HabitProgressRepository` progress
- Checkbox habit coins are `10 coins` through `EconomyConfig`

## Data Model

Events include both transient UI events and persistent timeline events:
- **RewardUiEvent**: Sealed class with data class subclasses for reward popups
- **GameEventEntity**: Persistent Room entity for the activity timeline
- **JournalEntryEntity**: Legacy Room entity retained for compatibility; no active journal generation system
- **AchievementEntity**: Persistent Room entity with unlock/claim state
- Various SharedFlow and StateFlow objects in ViewModels for UI state

## Known Gaps

1. **Limited Reward Event Types**: RewardUiEvent still has a fixed set of reward popup types, although timeline events are extensible through `GameEventType`.
2. **Event Data Constraints**: ChestReward.amount as Any limits type safety; mostly used for Int (coins) and String.
3. **Mixed Event Pipelines**: Reward UI events and persistent timeline events serve different purposes and should stay separated.
4. **Event History**: The activity timeline now persists major gameplay events; legacy `JournalEntryEntity` remains for compatibility but is no longer actively generated.
5. **UI Coupling**: Reward events are still tied to specific popups/screens; timeline events are more generic.
6. **Missing Gameplay Events**: Some actions such as habit creation/deletion, item purchases, and equipment changes are not yet timeline events.
7. **Event Timing**: Most events are logged immediately through async scopes; no delayed or scheduled timeline event system exists yet.
8. **Cross-Platform Consistency**: Event systems are Android/Kotlin-specific; no abstraction for potential cross-platform reuse.
