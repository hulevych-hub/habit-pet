# EVENTS

## Overview

Habit Pet uses an event-driven architecture to communicate between different game systems. Events are used to decouple components, trigger UI updates, generate journal entries, award rewards, and track player progress. The game implements multiple event systems for different purposes.

## Current Implementation

The event system consists of:
1. **RewardUiEvent** - Sealed class for reward popup events
2. **JournalEngine** - Generates journal entries based on pet/streak events
3. **AchievementEngine** - Monitors game state and unlocks achievements
4. **StreakEngine** - Evaluates streaks and awards chest rewards
5. **Reward Event Bus/Queue System** - Manages event prioritization and display
6. **Various ViewModel callbacks** - For UI state updates

## Rules

### RewardUiEvent System
The primary event system for rewards and popups:

**Event Types**:
- `CoinReward(amount: Int)` - Awards coins directly
- `LevelUpReward(level: Int, coins: Int)` - Awards level-up bonus coins
- `DragonEvolutionReward(fromStage: Int, toStage: Int)` - Signals evolution (0 coins)
- `StreakReward(streak: Int, coins: Int)` - Awards streak milestone coins
- `AchievementReward(achievementName: String, coins: Int)` - Awards achievement coins
- `ChestReward(rewardType: String, amount: Any)` - Awards chest contents (usually coins)

**Event Flow**:
1. Events are generated by various engines (AchievementEngine, StreakEngine, HabitDetailViewModel)
2. Events are added to RewardQueue via `rewardQueue.addReward(event)`
3. RewardQueue prioritizes events and emits them one at a time
4. RewardManager processes events, extracts coin values, and updates statistics
5. UI layer collects events and displays appropriate popups/screens

**Event Priority** (in RewardQueue):
1. LevelUpReward (priority 1 - highest)
2. DragonEvolutionReward (priority 2)
3. StreakReward (priority 3)
4. ChestReward (priority 4)
5. AchievementReward (priority 5 - lowest)

### JournalEngine Events
The JournalEngine creates permanent journal entries for significant milestones:

**Triggered Events**:
- **Pet Arrival**: When first journal entry is created (generateArrivalEntry)
- **Pet Evolution**: When pet.evolutionStage increases (generateEvolutionEntry)
- **Streak Milestones**: When streaks reach 7, 14, 30, 60, 100 days (generateStreakEntry)

**Journal Entry Content**:
- Arrival: "{pet.name} arrived."
- Evolution Stage 1: "{pet.name} hatched."
- Evolution Stage 2: "{pet.name} became a young dragon."
- Evolution Stage 3: "{pet.name} became an adult dragon."
- Evolution Stage 4: "{pet.name} became an ancient dragon."
- Streak: "Kept a {streak}-day streak!"

### AchievementEngine Events
Monitors five game aspects to unlock achievements:

**Observed Systems**:
- **Habit Count**: Unlocks "First Habit" when habits ≥ 1
- **Streaks**: Unlocks "7 Day Streak" (≥7) and "30 Day Streak" (≥30)
- **Completions**: Unlocks "100 Completions" when total completions ≥ 100
- **XP**: Unlocks "1000 XP" when pet XP ≥ 1000
- **Level**: Unlocks "Level 10" (≥10) and "Level 25" (≥25)

**Event Flow**:
1. AchievementEngine monitors respective repositories via Flow collectors
2. When threshold is met, checks if achievement is already unlocked
3. If not, updates AchievementEntity (isUnlocked = true, unlockedDate = timestamp)
4. Creates AchievementReward event with achievement name and rewardCoins
5. Adds event to RewardQueue for processing

### StreakEngine Events
Evaluates habit completion streaks and awards rewards:

**Trigger Conditions**:
- When all habits are completed on a given day
- Checks if currentStreak ≥ 7 AND currentStreak % 7 = 0 AND currentStreak > lastStreakAwardedAt

**Event Generated**:
- ChestReward with:
  * rewardType: "7_day_streak"
  * amount: 50 (coins as Int)
- Updates StatisticsEntity.lastStreakAwardedAt to currentStreak

### Additional Event Systems

**ViewModel State Updates**:
- HabitDetailViewModel emits:
  * _habitCompleted SharedFlow (Unit) when checkbox/timer habit completed
  * _rewardUiEvent SharedFlow (RewardUiEvent) for various rewards
  * _navigateBack SharedFlow (Unit) to navigate back after completion
- HomeScreenViewModel combines multiple states into UiState for UI consumption

**Database Change Events**:
- Room database flows notify observers of entity changes
- Used by ViewModels to update UI state when data changes

## Configuration

All event systems use hardcoded values:
- RewardUiEvent class definitions are sealed in RewardUiEvent.kt
- JournalEngine milestone array: [7, 14, 30, 60, 100]
- AchievementEngine thresholds: habit count ≥ 1, streaks ≥ 7/30, completions ≥ 100, XP ≥ 1000, levels ≥ 10/25
- StreakEngine chest award: 50 coins every 7 days
- HabitDetailViewModel level-up coins: level × 10
- HabitDetailViewModel level-up chest: 20 coins
- HabitDetailViewModel timer habit coins: (10 + sessionMinutes)
- HabitDetailViewModel checkbox habit coins: 1 (commented out in favor of XP-only)

## Data Model

Events are primarily transient objects:
- **RewardUiEvent**: Sealed class with data class subclasses
- **JournalEntryEntity**: Persistent Room entity for journal system
- **AchievementEntity**: Persistent Room entity with unlock state
- Various SharedFlow and StateFlow objects in ViewModels for UI state

## Known Gaps

1. **Limited Event Types**: RewardUiEvent has a fixed set of types; no extensible system for custom events.
2. **Event Data Constraints**: ChestReward.amount as Any limits type safety; mostly used for Int (coins) and String.
3. **Synchronous Processing**: Event handling occurs on ViewModel scopes; no async/event bus for cross-component communication beyond immediate processing.
4. **Event History**: No permanent log of events processed; only journal entries for specific milestones and achievements.
5. **UI Coupling**: Events are tightly coupled to specific UI popups/screens; no generic event handling system.
6. **Missing Gameplay Events**: No events for habit creation/deletion, item purchases, equipment changes, or other significant gameplay actions beyond rewards and milestones.
7. **Event Timing**: All events processed immediately; no delayed or scheduled event system.
8. **Cross-Platform Consistency**: Event systems are Android-specific; no abstraction for potential cross-platform reuse.
